# 第一部分：全局架构概览

## 1.1 问题定义

本比赛的核心任务是 **CTR/CVR 预测**（Click-Through Rate / Conversion Rate Prediction）：

* **输入**: 一条曝光请求，包含用户画像特征、物品画像特征、用户的多条历史行为序列、时间戳。
* **输出**: 一个 `[0, 1]` 之间的概率值，表示用户对该物品产生目标行为（如转化）的概率。
* **评估指标**: AUC（Area Under the ROC Curve）。AUC 越高越好。

数学上，我们要学习一个函数：

```
f(user_features, item_features, behavior_sequences, timestamp) → p ∈ [0, 1]
```

使得在所有测试样本上，AUC 最大化。

## 1.2 数据格式总览

每条样本是一个 Parquet 行，包含 7 列：

| 列名             | 类型              | 含义                                                  |
| -------------- | --------------- | --------------------------------------------------- |
| `user_id`      | `string`        | 用户唯一标识，如 `"user_2201"`                              |
| `item_id`      | `int`           | 物品唯一标识，如 `1836`                                     |
| `user_feature` | `array[struct]` | 用户静态画像特征列表                                          |
| `item_feature` | `array[struct]` | 物品静态画像特征列表                                          |
| `seq_feature`  | `struct`        | 包含多条行为序列（如 `action_seq`, `content_seq`, `item_seq`） |
| `timestamp`    | `int`           | 曝光请求时间戳（Unix 秒）                                     |
| `label`        | `array[struct]` | 标签数组，每个元素含 `action_time` 和 `action_type`            |

### 特征的嵌套结构

**`user_feature` / `item_feature`** 中的每个元素是一个 struct：

```json
{
  "feature_id": 6,
  "feature_value_type": "int_value",   // 可以是 int_value / float_value / int_array / float_array
  "int_value": 42,                      // 当 type 为 int_value 时有值
  "float_value": null,                  // 当 type 为 float_value 时有值
  "int_array": null,                    // 当 type 为 int_array 时有值
  "float_array": null                   // 当 type 为 float_array 时有值
}
```

**`seq_feature`** 是一个嵌套 struct：

```json
{
  "action_seq": [
    {"feature_id": 19, "feature_value_type": "int_array", "int_array": [1, 2, 3, 4, 5, 6]},
    {"feature_id": 20, "feature_value_type": "int_array", "int_array": [2, 2, 1, 3, 1, 4]}
  ],
  "content_seq": [...],
  "item_seq": [...]
}
```

**关键理解**: 每一条序列（如 `action_seq`）包含多个特征通道。每个通道是一个 `int_array`，其中第 `t` 个元素对应第 `t` 个时间步的该特征值。同一序列的所有通道共享相同的时间轴——`int_array[0]` 都是第 0 个事件，`int_array[1]` 都是第 1 个事件，以此类推。

## 1.3 模型设计哲学

UniScaleFormer 的设计基于三篇核心论文的深度融合与超越：

### 来自 InterFormer (CIKM 2025)

**核心思想**: 传统方法要么先做序列建模再做特征交互，要么反过来，都会造成「信息瓶颈」——来自一端的表征被过早压缩，无法充分与另一端交互。InterFormer 提出了「Interleaved」（交替）结构，让序列建模与特征交互在多个层之间交替进行。

**我们吸收了**: 交替更新机制——我们的 `UniScaleLayer` 每层都既做 cross-attention（序列→query）又做 TokenMixer（query↔static），实现信息的双向流动。

### 来自 OneTrans (arXiv 2025)

**核心思想**: 把所有异构特征——ID 特征、数值特征、序列特征——统一 tokenize 成一个 token 序列，然后用 **一个** Transformer 骨干统一处理。这个做法的好处是天然满足 Scale Law——只要增大 d_model 和 n_layers，模型容量就线性增长。

**我们吸收了**: 统一 tokenizer 设计——我们的 `UnifiedTokenizer` 确实把所有类型的特征都映射成同维度向量。但我们没有简单把所有 token 拼成一条流做 self-attention（那样在长序列时复杂度太高），而是分别编码后在 query 层面统一交互。

### 来自 HyFormer (arXiv 2026)

**核心思想**: 在多序列场景下，不同行为序列（如「点击序列」「搜索序列」「收藏序列」）蕴含不同维度的兴趣信号。简单地把它们拼起来当一条序列处理会丧失序列间的结构信息。HyFormer 证明了「分序列建模 + query-level 交互」效果更好。同时引入 memory token 对长序列降维。

**我们吸收了**: (1) 每条序列独立编码 + MemoryCompressor 压缩。(2) Query decoding 从多序列 memory 中提取信息。(3) Query boosting 让 query 与静态特征再交互。

### UniScaleFormer 的独创贡献

在上述三者基础上，UniScaleFormer 做了以下创新组合：

| 组件              | 灵感来源           | UniScaleFormer 的做法                           |
| --------------- | -------------- | -------------------------------------------- |
| 统一 Tokenization | OneTrans       | 所有静态特征 + 序列事件都经过统一 tokenizer                 |
| 分序列建模           | HyFormer       | 每条序列单独 self-attention + memory 压缩            |
| Memory 压缩       | HyFormer       | 可学习 memory token + cross-attention           |
| Query Decoding  | HyFormer       | Target-aware query 从 memory bank 提取          |
| 交替更新            | InterFormer    | UniScaleLayer 中 query↔static 交替 mix          |
| 显式交叉            | DeepFM / FM 理论 | FMHead 提供二阶显式特征交叉                            |
| Scale Law 友好    | OneTrans       | 增大 d_model / n_layers / num_queries 即可 scale |
| 辅助对比损失          | 自监督学习          | query 表征 vs item embedding 的对齐               |

## 1.5 与现有方法的对比及优势分析

### 方案一：传统 DIN / DIEN 类

```
序列 ──→ GRU/Attention ──→ 用户兴趣向量 ──→ concat(user_vec, item_vec) ──→ MLP
```

**劣势**:

* 序列建模和特征交互**完全分离**，无法在中间层交互。
* 多序列只能简单拼接或独立建模后 concat。
* 难以 scale：增大 GRU hidden 不能线性提升容量。

### 方案二：纯 Transformer (如 SASRec + 特征拼接)

```
all_features + all_seq_events ──→ 一条超长 token 流 ──→ Transformer ──→ [CLS]
```

**劣势**:

* 复杂度 O((S+L)²)，当序列很长（L=200+）且特征很多（S=100+）时代价极高。
* 无法区分不同序列的角色。
* 无法利用序列间结构信息。

### 方案三：UniScaleFormer (本方案)

```
静态特征 ──→ [S tokens] ──→ static encoder
多序列 ──→ 分别编码 ──→ memory compress ──→ [Ns*M memory tokens]
queries ──→ 交替(cross-attn memory + mix static) × N layers
最终 ──→ hybrid head (pooled + FM)
```

**优势**:

| 维度        | DIN/DIEN  | 纯 Transformer | **UniScaleFormer**    |
| --------- | --------- | ------------- | --------------------- |
| 信息流       | 单向(序列→特征) | 全双向但代价高       | **多层交替双向**            |
| 序列复杂度     | O(L)      | O((S+L)²)     | **O(L² + M²)** 其中 M≪L |
| 多序列感知     | ❌         | ❌             | **✅ 分序列建模**           |
| 显式交叉      | ❌         | ❌             | **✅ FM Head**         |
| Scale Law | ❌         | ✅             | **✅**                 |
| 时延控制      | ✅         | ❌(太慢)         | **✅ memory 压缩**       |

# 第三部分：utils/ 工具模块详解

## 3.1 utils/config.py

这是最简单的工具文件，只有两个函数。

### 函数：`load_config(path: str) → dict`

```
输入: path — YAML 配置文件的路径，如 "configs/base.yaml"
输出: dict — 解析后的配置字典

操作:
  1. 打开文件
  2. 用 yaml.safe_load() 解析
  3. 返回 Python dict
```

**为什么需要它**: 所有的超参数都集中在 YAML 文件中管理。训练、评估、推理脚本都通过此函数统一加载配置，避免硬编码。

### 函数：`ensure_dir(path: str) → None`

```
输入: path — 目录路径
输出: 无 (副作用：创建目录)

操作:
  1. 调用 os.makedirs(path, exist_ok=True)
```

**为什么需要它**: 训练时需要创建输出目录保存 checkpoint 和日志，此函数确保目录存在。

## 3.2 utils/metrics.py

### 函数：`binary_metrics(y_true, y_pred) → dict`

```
输入:
  - y_true: list[float] — 真实标签列表，每个值为 0.0 或 1.0
  - y_pred: list[float] — 模型预测概率列表，每个值在 [0, 1] 之间

输出: dict，包含:
  - "logloss": float — 交叉熵损失
  - "auc": float — ROC AUC 分数 (若只有一个类别则返回 nan)

操作:
  1. 将输入转成 numpy 数组
  2. 将预测值 clip 到 [1e-6, 1-1e-6] 防止 log(0) 溢出
  3. 调用 sklearn.metrics.log_loss 计算 logloss
  4. 检查是否有两个类别 → 若是，调用 roc_auc_score 计算 AUC
  5. 返回结果 dict
```

**设计决策**: AUC 是本比赛的核心指标。logloss 作为辅助参考指标，帮助判断模型是否在"校准"（calibration）层面也表现良好。

## 3.3 utils/training.py

### 函数：`set_seed(seed: int) → None`

```
输入: seed — 随机种子整数
输出: 无

操作:
  1. random.seed(seed)         — 设置 Python 标准库随机种子
  2. np.random.seed(seed)      — 设置 NumPy 随机种子
  3. torch.manual_seed(seed)   — 设置 PyTorch CPU 随机种子
  4. torch.cuda.manual_seed_all(seed)  — 设置所有 GPU 随机种子
```

**为什么需要**: 可复现性是科研和比赛的基本要求。相同 seed 应产生相同结果。

### 函数：`move_to_device(x, device) → 同类型`

```
输入:
  - x: 任意类型 — 可以是 Tensor、dict、list 或其他
  - device: torch.device — 目标设备 (如 cuda:0)

输出: 与输入相同的结构，但所有 Tensor 都已移动到目标设备

操作:
  1. 若 x 是 Tensor: 返回 x.to(device)
  2. 若 x 是 dict: 递归处理每个 value
  3. 若 x 是 list: 递归处理每个元素
  4. 否则: 原样返回
```

**为什么需要**: collate 函数返回的 batch 是嵌套 dict，需要一次性把所有张量移到 GPU。

### 函数：`cosine_lr(step, total, warmup, base_lr, min_lr) → float`

```
输入:
  - step: int — 当前全局步数 (从 0 开始)
  - total: int — 总步数 (epochs × batches_per_epoch)
  - warmup: int — 线性预热步数
  - base_lr: float — 峰值学习率
  - min_lr: float — 最低学习率

输出: float — 当前步应使用的学习率

操作:
  1. 若 step < warmup:
       lr = base_lr × (step + 1) / warmup   ← 线性预热
  2. 否则:
       progress = (step - warmup) / (total - warmup)
       cosine = 0.5 × (1 + cos(π × progress))
       lr = min_lr + (base_lr - min_lr) × cosine  ← 余弦退火
  3. 返回 lr
```

**学习率曲线示意**:

```
lr ↑
   │     ╱‾‾‾‾‾‾‾‾╲
   │    ╱            ╲
   │   ╱              ╲
   │  ╱                ╲________
   │ ╱  warmup    cosine decay
   └───────────────────────────→ step
```

**为什么用余弦退火**: (1) 收敛更稳定; (2) 末期学习率平滑降低，不会突然震荡; (3) 是当前 NLP/CV/Rec 领域的标准做法。

## 3.4 utils/parquet_dataset.py

这个文件负责**从磁盘上的 Parquet 文件中高效读取数据**。它解决了一个核心问题：Parquet 文件通常很大，不能一次性全部读入内存，但 PyTorch 的 Dataset 需要通过 `__getitem__(idx)` 随机访问任意一行。

### 类：`IndexedParquetDataset(Dataset)`

#### `__init__(self, paths, limit=None)`

```
输入:
  - paths: list[str] — Parquet 文件路径列表
  - limit: int or None — 最多读取多少行 (用于调试)

操作:
  1. 对每个文件创建 pq.ParquetFile 对象 (不读数据，只读元信息)
  2. 遍历每个文件的每个 row group:
     - 从元信息中获取该 row group 的行数
     - 为每一行创建一个 (file_idx, row_group_idx, row_idx) 三元组
     - 存入 self.index 列表
  3. 若设置了 limit，到达上限后立即停止

存储:
  - self.files: list[ParquetFile]  — 文件句柄
  - self.index: list[tuple(int,int,int)] — 全局行索引
```

#### `__len__(self) → int`

```
返回 self.index 的长度，即总行数。
```

#### `_read_rg(self, fi, rg) → list[dict]`

```
输入: fi — 文件索引, rg — row group 索引
输出: list[dict] — 该 row group 的所有行，每行是一个 dict

操作:
  1. 调用 self.files[fi].read_row_group(rg) 读取整个 row group
  2. .to_pylist() 转为 Python dict 列表
  3. 结果被 @lru_cache(maxsize=8) 缓存

注意: @lru_cache 的作用极为关键。同一个 row group 的多行在短时间内
被连续访问时，只需读一次磁盘。maxsize=8 表示最多同时缓存 8 个 row group。
```

#### `__getitem__(self, idx) → dict`

```
输入: idx — 全局行索引
输出: dict — 该行的所有列，如 {'item_id': 1836, 'user_id': 'user_2201', ...}

操作:
  1. 从 self.index[idx] 取出 (fi, rg, ri) 三元组
  2. 调用 self._read_rg(fi, rg) 获取该 row group 的所有行
  3. 返回第 ri 行
```

### 类：`TAACDataset(Dataset)`

这是对 `IndexedParquetDataset` 的高层封装，增加了标签解析逻辑。

#### `__init__(self, data_dir, split, positive_action_types, limit=None, file_pattern='*.parquet')`

```
输入:
  - data_dir: str — 数据根目录，如 "./data"
  - split: str — 数据集划分，如 "train" / "valid" / "test"
  - positive_action_types: list[int] — 被视为正样本的 action_type 集合
  - limit: int or None — 最多读取行数
  - file_pattern: str — 文件匹配模式

操作:
  1. 先尝试 data_dir/split/ 目录下的 *.parquet
  2. 若不存在，尝试 data_dir/split*.parquet
  3. 创建 IndexedParquetDataset
  4. 存储 positive_action_types 为集合
```

#### `__getitem__(self, idx) → dict`

```
输入: idx — 行索引
输出: dict — 包含原始数据 + 额外的 'target' 字段

操作:
  1. 从底层 dataset 获取原始行
  2. 解析 label 字段:
     - 遍历 label 数组中的每个元素
     - 若任何元素的 action_type 在 positive_action_types 中 → target = 1.0
     - 否则 → target = 0.0
  3. 返回包含 target 的完整 dict
```

**标签解析逻辑示例**:

```
label = [{"action_time": 1770700819, "action_type": 1}]
positive_action_types = {1}

→ 找到 action_type=1 在集合中 → target = 1.0
```

```
label = [{"action_time": 1770700819, "action_type": 0}]
positive_action_types = {1}

→ 没有匹配 → target = 0.0
```

```
label = []   (测试集)
→ 没有任何元素 → target = 0.0
```

## 3.5 utils/collate.py

**这是整个项目中最复杂的工具文件**。它负责将一组原始 Python dict (来自 TAACDataset) 转换成模型可以直接消费的**固定形状张量 batch**。

### 全局常量

```python
TYPE_TO_ID = {
    'unknown': 0,
    'int_value': 1,
    'float_value': 2,
    'int_array': 3,
    'float_array': 4,
    'int_array_and_float_array': 5
}
```

作用: 将特征值类型名称映射为整数 ID，用于 type embedding。

```python
SEQ_NAME_TO_ID = {'action_seq': 1, 'content_seq': 2, 'item_seq': 3}
```

作用: 将序列名称映射为整数 ID，用于 sequence type embedding。

```python
SPECIAL = {'user_id': 1, 'item_id': 2, 'timestamp_bucket': 3}
```

作用: 为特殊 token（user_id、item_id、时间桶）分配 feature_id。

### 辅助函数：`_h(s: str, mod: int) → int`

```
输入:
  - s: 任意字符串
  - mod: 哈希桶大小

输出: 1 到 mod-1 之间的整数（注意避开 0，因为 0 是 padding）

操作:
  1. 计算 MD5 哈希
  2. 取前 12 个十六进制字符 → 转为 48-bit 整数
  3. 模 (mod-1) + 1

为什么是 MD5: 速度 > 安全性，且在百万级桶中冲突率足够低。
为什么 +1: embedding 的 padding_idx=0，有效 ID 从 1 开始。
```

### 类：`TAACCollator`

#### `__init__(self, cfg)`

```
输入: cfg — 完整配置 dict

操作: 从配置中提取并存储以下参数:
  - self.hash_size     — 哈希 embedding 表大小 (如 4000007)
  - self.max_static    — 最大静态 token 数 (如 128)
  - self.max_float     — 最大浮点维度 (如 64)
  - self.max_seq       — 最大序列长度 (如 128)
  - self.max_event     — 每个序列事件的最大特征数 (如 8)
  - self.sequence_names — 要处理的序列名列表
```

#### 方法：`_float_vals(self, feat) → list[float]`

```
输入: feat — 一个特征 struct dict

输出: list[float]，长度恒为 self.max_float

操作:
  1. 若 feat 有 float_array → 取前 max_float 个值
  2. 否则若有 float_value → 包装成单元素列表
  3. 否则 → 空列表
  4. 不足 max_float 的部分用 0.0 填充

示例:
  feat = {"float_array": [0.1, 0.2, 0.3], ...}, max_float = 5
  → [0.1, 0.2, 0.3, 0.0, 0.0]
```

#### 方法：`_stat_tok(self, ns, feat) → tuple(token_id, feature_id, type_id, float_values)`

```
输入:
  - ns: str — 命名空间，"user" 或 "item"
  - feat: dict — 特征 struct

输出: (token_id: int, feature_id: int, type_id: int, float_values: list[float])

操作:
  1. 提取 feature_id 和 feature_value_type
  2. 根据值类型构造哈希 key:
     - int_value → str(int_value)
     - int_array → 前 8 个元素用逗号拼接
     - float 类型 → 固定字符串 "float"
     - 无值 → "null"
  3. token_id = _h(f'{ns}|{fid}|{type}|{key}', hash_size)
  4. type_id = TYPE_TO_ID[type_name]
  5. float_values = self._float_vals(feat)

示例:
  ns="user", feat={"feature_id":58, "feature_value_type":"int_value", "int_value":5}
  → key = "user|58|int_value|5"
  → token_id = _h(key, 4000007)  → 某个 1~4000006 的整数
  → feature_id = 58
  → type_id = 1 (int_value)
  → float_values = [0.0, 0.0, ..., 0.0]
```

**为什么用哈希而不是维护 vocabulary**:

* 比赛数据量巨大，特征值集合未知且可能很大
* 哈希 embedding 不需要预先遍历数据统计 vocabulary
* 代价是极小的哈希冲突率，可通过增大 hash_size 降低

#### 方法：`_special(self, name, val) → tuple`

```
输入:
  - name: str — "user_id" / "item_id" / "timestamp_bucket"
  - val: Any — 对应值

输出: (token_id, feature_id, type_id, float_values)

操作:
  1. token_id = _h(f'{name}|{val}', hash_size)
  2. feature_id = SPECIAL[name] (1/2/3)
  3. type_id = 0 (特殊类型)
  4. float_values = 全零向量
```

#### 方法：`_build_static(self, sample) → tuple`

这是将一条样本的**所有静态特征**转换成固定长度张量的核心函数。

```
输入: sample — 一条样本 dict

输出: (ids, fids, tids, fvals, mask) 五元组
  - ids:   [max_static] LongTensor  — 哈希后的 token ID
  - fids:  [max_static] LongTensor  — 原始 feature ID
  - tids:  [max_static] LongTensor  — 类型 ID
  - fvals: [max_static, max_float] FloatTensor — 浮点值向量
  - mask:  [max_static] BoolTensor  — True 表示有效位置

操作:
  1. 构建 token 列表 (顺序如下):
     位置 0:  user_id 特殊 token
     位置 1:  item_id 特殊 token     ← 注意：item_id 固定在位置 1
     位置 2:  timestamp_bucket 特殊 token
     位置 3+: user_feature 中的每个特征
     位置 N+: item_feature 中的每个特征
  2. 截断到 max_static 长度
  3. 填入对应的固定长度张量
  4. 不足的位置自动为 0 (padding)

为什么 item_id 固定在位置 1:
  辅助对比损失需要知道 item 的 embedding，通过 batch['static_token_ids'][:,1]
  直接索引位置 1 来获取 item_id 的 token_id。
```

#### 方法：`_build_seq(self, name, payload) → tuple`

这是将一条行为序列转换成固定形状张量的函数。

```
输入:
  - name: str — 序列名，如 "action_seq"
  - payload: list[dict] — 该序列的特征通道列表

输出: (ids, fids, tids, pos, mask, sid) 六元组
  - ids:  [max_seq, max_event] LongTensor  — 每个事件中每个特征的 token ID
  - fids: [max_seq, max_event] LongTensor  — feature ID
  - tids: [max_seq, max_event] LongTensor  — type ID
  - pos:  [max_seq] LongTensor             — 位置索引 (0, 1, 2, ...)
  - mask: [max_seq] BoolTensor             — True 表示该位置有事件
  - sid:  [max_seq] LongTensor             — 序列类型 ID (常量)

操作:
  1. 遍历所有特征通道，找到最长的 int_array 长度 → 实际序列长度 L
  2. L = min(L, max_seq) → 超长序列截断
  3. 若 L > max_seq: 取最后 max_seq 个事件 (保留最近行为)
  4. 对每个输出位置 o (对应源位置 p):
     - 遍历每个特征通道
     - 若该通道的 int_array[p] 存在:
       - 计算 token_id = _h(f'seq:{name}|{fid}|{type}|{value}', hash_size)
       - 存入 ids[o, k], fids[o, k], tids[o, k]
       - k += 1 (最多 max_event 个)
     - 若 k > 0: mask[o] = True

示例:
  name = "action_seq"
  payload = [
    {"feature_id": 19, "int_array": [1, 2, 3]},   ← 特征通道 1
    {"feature_id": 20, "int_array": [5, 6, 7]},   ← 特征通道 2
  ]
  max_seq = 4, max_event = 8

  输出张量 ids 的有效部分:
    ids[0] = [hash("seq:action_seq|19|int_array|1"), hash("seq:action_seq|20|int_array|5"), 0, ...]
    ids[1] = [hash("seq:action_seq|19|int_array|2"), hash("seq:action_seq|20|int_array|6"), 0, ...]
    ids[2] = [hash("seq:action_seq|19|int_array|3"), hash("seq:action_seq|20|int_array|7"), 0, ...]
    ids[3] = [0, 0, 0, ...]  ← padding
  mask = [True, True, True, False]

为什么取「最后 max_seq 个事件」而不是「前 max_seq 个」:
  用户最近的行为比远古行为更能预测当前意图。这是工业推荐系统的标准做法。
```

#### 方法：`__call__(self, batch) → dict`

**这是 DataLoader 实际调用的入口函数**。

```
输入: batch — list[dict]，长度 B，每个元素是 TAACDataset.__getitem__ 的返回值

输出: dict[str, Tensor]，包含以下键:

  {
    'static_token_ids':    [B, max_static]                 LongTensor
    'static_feature_ids':  [B, max_static]                 LongTensor
    'static_type_ids':     [B, max_static]                 LongTensor
    'static_float_values': [B, max_static, max_float]      FloatTensor
    'static_mask':         [B, max_static]                 BoolTensor
    'seq_token_ids':       [B, Ns, max_seq, max_event]     LongTensor
    'seq_feature_ids':     [B, Ns, max_seq, max_event]     LongTensor
    'seq_type_ids':        [B, Ns, max_seq, max_event]     LongTensor
    'seq_pos_ids':         [B, Ns, max_seq]                LongTensor
    'seq_mask':            [B, Ns, max_seq]                BoolTensor
    'seq_name_ids':        [B, Ns, max_seq]                LongTensor
    'targets':             [B]                             FloatTensor
    'item_ids':            [B]                             LongTensor
  }

  其中 Ns = len(sequence_names) = 3 (action_seq, content_seq, item_seq)

操作:
  1. 对每个样本:
     a. 调用 _build_static() → 5 个静态张量
     b. 对每个 sequence_name 调用 _build_seq() → 6 个序列张量
     c. 将 3 个序列的张量 stack → [3, max_seq, ...]
  2. 把所有样本的张量 stack → [B, ...]
  3. 提取 targets 和 item_ids
```

# 第四部分：models/ 模型模块详解

## 4.1 models/tokenizer.py — UnifiedTokenizer

### 设计理念

推荐系统的特征天然是**异构的**：

* ID 特征 (user_id, item_id) → 离散，需要 embedding
* 整数特征 (int_value) → 离散但基数可能很大
* 浮点数组 (float_array) → 连续，需要投影
* 序列事件 → 由多个 ID 组成

传统做法是为每种类型单独设计 encoder。UniScaleFormer 的 UnifiedTokenizer 将它们**统一映射到相同维度 d_model**，使得后续的 Transformer 层无需区分特征来源。

### 类：`UnifiedTokenizer(nn.Module)`

#### `__init__(...)`

```
输入参数:
  - hash_size: int — 哈希 embedding 表大小 (如 4000007)
  - feature_vocab_size: int — feature_id embedding 表大小 (如 16384)
  - type_vocab_size: int — 类型 embedding 表大小 (如 16)
  - seq_vocab_size: int — 序列类型 embedding 表大小 (如 16)
  - max_position: int — 最大位置编码 (max_seq_len + 8)
  - d_model: int — 输出向量维度 (如 192)
  - max_float_dim: int — 浮点输入维度 (如 64)
  - dropout: float — dropout 率

创建的子模块:
  1. token_emb:   Embedding(hash_size+1, d_model, padding_idx=0)
     → 将哈希后的 token ID 映射为向量
     → padding_idx=0 确保 padding 位置的 embedding 恒为零向量

  2. feature_emb: Embedding(feature_vocab_size, d_model, padding_idx=0)
     → 将 feature_id 映射为向量
     → 同一特征的不同值共享同一个 feature embedding

  3. type_emb:    Embedding(type_vocab_size, d_model, padding_idx=0)
     → 将特征类型 (int_value/float_array/...) 映射为向量

  4. seq_emb:     Embedding(seq_vocab_size, d_model, padding_idx=0)
     → 将序列类型 (action_seq/content_seq/item_seq) 映射为向量

  5. pos_emb:     Embedding(max_position, d_model)
     → 为序列中的每个位置提供位置编码

  6. float_proj:  LayerNorm(max_float_dim) → Linear → GELU → Linear
     → 将浮点特征值投影到 d_model 维
     → 两层 MLP 比单层线性更有表达力
     → LayerNorm 帮助稳定不同量级的浮点输入
```

#### 方法：`encode_static(token_ids, feature_ids, type_ids, float_values) → Tensor`

```
输入:
  - token_ids:    [B, S] LongTensor  — 哈希后的 token ID
  - feature_ids:  [B, S] LongTensor  — 原始 feature ID
  - type_ids:     [B, S] LongTensor  — 类型 ID
  - float_values: [B, S, F] FloatTensor — 浮点特征值

输出: [B, S, D] FloatTensor — 每个静态 token 的 embedding

操作:
  x = token_emb(token_ids)       # [B, S, D] — 主要语义信息
    + feature_emb(feature_ids)    # [B, S, D] — "这是第几号特征"
    + type_emb(type_ids)          # [B, S, D] — "这是什么类型"
    + float_proj(float_values)    # [B, S, D] — 浮点值的投影
  return dropout(x)

数学公式:
  h_i = E_token(t_i) + E_feature(f_i) + E_type(τ_i) + MLP(v_i)

为什么是「加法」而不是「拼接」:
  - 拼接会导致维度爆炸 (4*D → 需要投影回 D)
  - 加法是 Transformer 位置编码的标准做法 (position + token embedding)
  - 经验上，加法在推荐场景中效果不逊于拼接，且计算量更小

为什么包含 float_proj:
  - 对于 int_value 类特征，float_values 全为 0 → float_proj 输出接近零向量 → 不影响
  - 对于 float_array 类特征，token_emb 只提供"这是个浮点特征"的类型信息，
    float_proj 提供实际数值信息 → 两者互补
```

#### 方法：`encode_sequence_events(token_ids, feature_ids, type_ids, pos_ids, seq_name_ids) → Tensor`

```
输入:
  - token_ids:    [B, Ns, L, F] LongTensor — 每个事件中每个特征的 token ID
  - feature_ids:  [B, Ns, L, F] LongTensor
  - type_ids:     [B, Ns, L, F] LongTensor
  - pos_ids:      [B, Ns, L]    LongTensor — 位置索引
  - seq_name_ids: [B, Ns, L]    LongTensor — 序列类型 ID

  其中: B=batch, Ns=序列数(3), L=max_seq_len, F=max_event_features

输出: [B, Ns, L, D] FloatTensor — 每条序列中每个事件的 embedding

操作:
  1. token = token_emb(token_ids) + feature_emb(feature_ids) + type_emb(type_ids)
     → [B, Ns, L, F, D]  — 每个特征维度独立 embedding

  2. feat_mask = (token_ids ≠ 0).float().unsqueeze(-1)
     → [B, Ns, L, F, 1]  — 标记哪些特征位置是有效的

  3. event_repr = (token × mask).sum(dim=3) / mask.sum(dim=3).clamp_min(1)
     → [B, Ns, L, D]  — 对每个事件的多个特征取 masked mean pooling

  4. event_repr += pos_emb(pos_ids) + seq_emb(seq_name_ids)
     → [B, Ns, L, D]  — 加上位置编码和序列类型编码

  5. return dropout(event_repr)

为什么对事件内特征做 mean pooling:
  - 一个事件（如一次点击）通常对应 2-5 个特征（物品 ID、类别、时长等）
  - 这些特征描述的是同一个事件，应该聚合成一个事件表征
  - Mean pooling 是最简单高效的聚合方式
  - 后续的 self-attention 会在事件之间建模依赖关系

为什么需要 seq_emb:
  - 不同序列代表不同类型的行为 (点击 vs 搜索 vs 收藏)
  - seq_emb 让模型知道当前 token 来自哪条序列
  - 这在后续 memory compressor 中尤为重要
```

## 4.2 models/blocks.py — 所有基础构建块

### 类：`RMSNorm(nn.Module)`

**Root Mean Square Layer Normalization**

```
初始化:
  - dim: int — 输入向量维度
  - eps: float = 1e-6 — 防止除零的小常数
  - 可学习参数: self.weight [dim] — 缩放因子，初始化为全 1

forward(x: [*, D]) → [*, D]:
  1. norm = mean(x², dim=-1, keepdim=True)
  2. x_normalized = x / √(norm + eps)
  3. return x_normalized × self.weight

数学公式:
  RMSNorm(x) = x / RMS(x) × γ
  其中 RMS(x) = √(mean(x²))

与 LayerNorm 的区别:
  - LayerNorm: 先减均值再除标准差，有两个可学习参数 (γ, β)
  - RMSNorm: 不减均值，只除 RMS，只有一个可学习参数 (γ)
  - RMSNorm 计算量更小 (省去 mean 计算)，效果相当
  - 是 LLaMA / Mistral 等现代架构的标准选择
```

### 类：`SwiGLU(nn.Module)`

**Swish-Gated Linear Unit**，来自 Google 的 PaLM 论文。

```
初始化:
  - dim: int — 输入/输出维度
  - hidden_dim: int — 中间隐藏维度 (通常 4 × dim)
  - dropout: float

  子模块:
    w1: Linear(dim → hidden_dim)   — 主路径
    w2: Linear(dim → hidden_dim)   — 门控路径
    out: Linear(hidden_dim → dim)  — 输出投影
    dropout: Dropout(p)

forward(x: [*, D]) → [*, D]:
  gate = SiLU(w1(x))      # 主路径 + Swish 激活
  value = w2(x)             # 门控路径 (无激活)
  hidden = gate × value     # 逐元素相乘 → 门控
  return out(dropout(hidden))

数学公式:
  SwiGLU(x) = (SiLU(xW₁) ⊙ xW₂)W_out

为什么用 SwiGLU 而不是普通的 ReLU MLP:
  - SwiGLU 在多个 NLP benchmark 上显著优于 ReLU/GELU MLP
  - 门控机制让网络能自适应地选择传递哪些信息
  - 代价：多了一个 Linear 层 (w2)，但维度可以调小来补偿
```

### 类：`SimpleMHA(nn.Module)`

**Simple Multi-Head Attention**，手写实现而非使用 `nn.MultiheadAttention`。

```
初始化:
  - d_model: int — 输入维度
  - n_heads: int — 注意力头数
  - dropout: float

  断言: d_model % n_heads == 0
  head_dim = d_model // n_heads

  子模块:
    q_proj: Linear(d_model → d_model)  — Query 投影
    k_proj: Linear(d_model → d_model)  — Key 投影
    v_proj: Linear(d_model → d_model)  — Value 投影
    out_proj: Linear(d_model → d_model) — 输出投影
    dropout: Dropout(p)

forward(q, k, v, kv_mask=None):

  输入:
    q: [B, Q, D]     — Query
    k: [B, K, D]     — Key
    v: [B, K, D]     — Value
    kv_mask: [B, K] BoolTensor or None — True 表示有效位置

  输出: [B, Q, D]

  操作:
    1. Q = q_proj(q).reshape(B, Q, H, H_dim).transpose(1,2)  → [B, H, Q, H_dim]
    2. K = k_proj(k).reshape(B, K, H, H_dim).transpose(1,2)  → [B, H, K, H_dim]
    3. V = v_proj(v).reshape(B, K, H, H_dim).transpose(1,2)  → [B, H, K, H_dim]
    4. scores = Q @ K^T / √(H_dim)                            → [B, H, Q, K]
    5. 若有 mask: scores[~mask] = -10000 (使 softmax 后接近 0)
    6. attn = softmax(scores, dim=-1)                          → [B, H, Q, K]
    7. attn = dropout(attn)
    8. out = (attn @ V).transpose(1,2).reshape(B, Q, D)       → [B, Q, D]
    9. return out_proj(out)

为什么不用 nn.MultiheadAttention:
  - nn.MHA 在某些 PyTorch 版本的 CPU 上反向传播极慢 (已知问题)
  - 手写版本行为完全透明，方便调试和扩展
  - 性能上在 GPU 上几乎无差异

注意力机制的直觉:
  - Q 是"我在找什么"
  - K 是"我有什么"
  - V 是"我能提供什么"
  - scores = Q·K 衡量"我找的和你有的匹配度"
  - attn = softmax(scores) 归一化为概率分布
  - output = attn·V 加权平均所有 V
```

### 类：`CrossAttentionBlock(nn.Module)`

**带残差连接的 Cross-Attention。**

```
初始化:
  子模块:
    norm_q: RMSNorm(d_model)   — 对 Query 归一化
    norm_kv: RMSNorm(d_model)  — 对 Key/Value 归一化
    attn: SimpleMHA(d_model, n_heads, dropout)
    drop: Dropout(p)

forward(q, kv, kv_mask=None):

  输入:
    q:  [B, Q, D]     — Query 序列 (如 memory tokens)
    kv: [B, K, D]     — Key/Value 序列 (如序列事件)
    kv_mask: [B, K]   — Key/Value 的有效性 mask

  输出: [B, Q, D]

  操作:
    residual = q
    q_norm = RMSNorm(q)
    kv_norm = RMSNorm(kv)
    out = MHA(q_norm, kv_norm, kv_norm, kv_mask)
    return residual + dropout(out)

  这就是标准的 Pre-Norm Residual Cross-Attention:
    x ← x + Dropout(CrossAttn(Norm(x), Norm(kv)))

为什么是 Pre-Norm (先归一化再 attention):
  - Pre-Norm 比 Post-Norm 训练更稳定，不需要 careful weight init
  - 是 GPT-2 以来的标准做法
  - 特别适合深层网络，梯度传播更通畅
```

### 类：`SelfAttentionBlock(nn.Module)`

**带残差连接的 Self-Attention。**

```
forward(x, mask=None):

  输入:
    x:    [B, L, D]   — 输入序列
    mask: [B, L]      — 有效性 mask

  输出: [B, L, D]

  操作:
    y = RMSNorm(x)
    out = MHA(y, y, y, mask)   ← Q=K=V=y，即 self-attention
    return x + dropout(out)

与 CrossAttentionBlock 的唯一区别:
  - Q、K、V 都来自同一个输入 x (self = 自我注意)
  - 只有一个 norm (因为 Q/K/V 共享输入)
```

### 类：`FeedForwardBlock(nn.Module)`

**带残差连接的前馈网络。**

```
forward(x: [B, L, D]) → [B, L, D]:

  操作:
    return x + dropout(SwiGLU(RMSNorm(x)))

  展开:
    h = RMSNorm(x)
    gate = SiLU(h @ W1)
    value = h @ W2
    ff_out = (gate ⊙ value) @ W_out
    return x + dropout(ff_out)

一个标准的 Transformer 块 = Self-Attention + FeedForward:
  SelfAttentionBlock 处理 token 间关系 ("看别人")
  FeedForwardBlock 处理每个 token 自身的表征 ("思考自己")
```

### 类：`TokenMixer(nn.Module)`

**在 token 维度上做 MLP mixing。** 灵感来自 MLP-Mixer。

```
初始化:
  - d_model: int — 通道维度
  - num_tokens: int — token 数量 (必须固定)
  - expansion: int = 2 — 隐藏层扩展倍数

  子模块:
    norm: RMSNorm(d_model)
    fc1: Linear(num_tokens → num_tokens * expansion)
    fc2: Linear(num_tokens * expansion → num_tokens)

forward(x: [B, T, D]) → [B, T, D]:

  操作:
    y = RMSNorm(x)              → [B, T, D]
    y = y.transpose(1, 2)       → [B, D, T]   ← 把 token 维放到最后
    y = fc2(GELU(fc1(y)))       → [B, D, T]   ← 在 token 维上做 MLP
    y = y.transpose(1, 2)       → [B, T, D]   ← 恢复原始维度
    return x + dropout(y)

为什么需要 TokenMixer:
  - Self-Attention 在 token 间做依赖建模，但计算量 O(T²D)
  - TokenMixer 用线性层在 token 维度混合，计算量 O(TD·T) = O(T²D)，量级相同，但常数更小
  - 关键是: TokenMixer 把 query tokens 和 static tokens **拼在一起**做 mixing，
    实现了 query 与 static 之间的信息交换 (即 InterFormer 的"交替更新"效果)
  - 这是 UniScaleFormer 的核心创新之一

直觉解释:
  Self-Attention: "我仔细看看每个 token 跟我的关系" (data-dependent)
  TokenMixer: "不管内容是什么，按固定模式混合所有 token" (data-independent)
  两者互补: 一个灵活一个高效
```

### 类：`MemoryCompressor(nn.Module)`

**将长序列压缩成少量 memory token。** 这是处理长行为序列的关键组件。

```
初始化:
  - d_model: int
  - memory_tokens: int — memory token 数量 (如 12)
  - n_heads: int
  - dropout: float

  子模块:
    memory: Parameter [1, M, D] — 可学习的 memory token 初始值
    cross: CrossAttentionBlock
    ff: FeedForwardBlock

forward(x, mask):

  输入:
    x:    [B, L, D]   — 序列编码后的 token (L 可能很大，如 128)
    mask: [B, L]      — 有效性 mask

  输出: [B, M, D]    — 压缩后的 memory token (M 很小，如 12)

  操作:
    mem = self.memory.expand(B, -1, -1)    → [B, M, D]   ← 广播到 batch
    mem = CrossAttn(q=mem, kv=x, mask)     → [B, M, D]   ← memory 向序列查询
    mem = FeedForward(mem)                 → [B, M, D]   ← 非线性变换
    return mem

原理:
  - M 个 memory token 就像 M 个"问题"
  - 通过 cross-attention，每个 memory token 从长序列中"检索"相关信息
  - 结果是 M 个"答案"，每个都聚焦于序列的不同方面
  - 从 L 个 token 压缩到 M 个 token (L=128 → M=12)

为什么叫"可学习的 memory token":
  - 初始值是随机参数，通过训练自动学会"问什么问题"
  - 不同的 memory token 会学会关注序列的不同模式
  - 如 token 1 关注最近行为，token 2 关注高频物品，token 3 关注类别分布...

复杂度分析:
  - 直接对长序列做 self-attention: O(L²D) = O(128²×192) ≈ 3M
  - Memory 压缩后: O(M·L·D) + O(M²·D) ≈ O(12×128×192) ≈ 300K
  - 后续层只在 M 个 memory token 上操作，进一步节省
```

## 4.3 models/uniscaleformer.py — 核心模型

### 类：`UniScaleLayer(nn.Module)`

**UniScaleFormer 的一个层。每层执行「查询-交叉-混合-前馈」四步。**

```
初始化:
  - d_model, n_heads, num_mix_tokens, dropout
  子模块:
    cross: CrossAttentionBlock ← query 从 memory bank 提取序列信息
    mix:   TokenMixer          ← query 和 static token 之间交换信息
    ff:    FeedForwardBlock    ← 非线性变换

forward(queries, kv, kv_mask, static_tokens):

  输入:
    queries:       [B, Q, D]    — 当前层的 query token
    kv:            [B, K, D]    — memory bank (所有序列的 memory 拼接)
    kv_mask:       [B, K]       — memory 有效性 mask
    static_tokens: [B, S, D]    — 静态特征 token

  输出: [B, Q, D] — 更新后的 query token

  操作:
    1. q = CrossAttn(queries, kv, kv_mask)   → [B, Q, D]
       ← query 从 memory bank 中提取用户历史兴趣信息

    2. x = cat([q, static_tokens], dim=1)    → [B, Q+S, D]
       ← 把 query 和 static token 拼接

    3. x = FeedForward(TokenMixer(x))        → [B, Q+S, D]
       ← 在拼接后的 token 上做混合 + 非线性变换
       ← 这一步让 query 和 static token 互相交换信息

    4. return x[:, :Q]                       → [B, Q, D]
       ← 只取回 query 部分 (static token 不更新)

关键设计决策:
  - static_tokens 在所有层中共享同一份，不更新。
    这是因为静态特征本身不需要多层精炼，
    但 query 需要在每一层吸收越来越丰富的上下文。
  - TokenMixer 让 query 能"看到" static token，
    CrossAttention 让 query 能"看到" 序列 memory。
    两者交替进行 = InterFormer 的核心思想。
```

### 类：`FMHead(nn.Module)`

**Factorization Machine 二阶交叉头。**

```
forward(x: [B, T, D]) → [B, 1]:

  操作:
    summed = x.sum(dim=1)                  → [B, D]     ← 所有 token 的嵌入之和
    summed_square = summed²                → [B, D]     ← (Σ xᵢ)²
    square_summed = (x²).sum(dim=1)        → [B, D]     ← Σ (xᵢ²)
    output = 0.5 × (summed_square - square_summed).sum(dim=-1, keepdim=True)
    → [B, 1]

  数学公式 (FM 二阶交叉):
    FM₂(x₁, ..., xₙ) = Σᵢ<ⱼ <xᵢ, xⱼ>
                       = 0.5 × (‖Σᵢxᵢ‖² - Σᵢ‖xᵢ‖²)

  这利用了恒等式:
    Σᵢ<ⱼ <xᵢ,xⱼ> = 0.5 × [(Σᵢxᵢ)·(Σⱼxⱼ) - Σᵢ(xᵢ·xᵢ)]

为什么加 FM Head:
  - Transformer 的 self-attention 是隐式交互：token 间通过注意力权重交互
  - FM 是显式交互：直接计算所有 token 对的内积
  - 两者互补：显式交叉能捕获 Transformer 难以学到的低阶交互模式
  - 在 DeepFM 论文中已被证明有效
  - 计算量仅 O(TD)，几乎不增加开销
```

### 类：`UniScaleFormer(nn.Module)` — 主模型

#### `__init__(self, cfg)`

```
输入: cfg — 完整配置 dict

创建的子模块一览:

  ┌──────────────────────────────────────────────────────┐
  │ 1. self.tokenizer: UnifiedTokenizer                  │
  │    → 将所有特征统一 tokenize                          │
  │                                                      │
  │ 2. self.static_encoder: ModuleList[SelfAttnBlock]    │
  │    → 对静态 token 做 self-attention (默认 1 层)       │
  │                                                      │
  │ 3. self.seq_encoder: ModuleList[SelfAttnBlock]       │
  │    → 对每条序列做 self-attention (默认 1 层)          │
  │    → 注意：3 条序列共享同一组 seq_encoder 参数        │
  │                                                      │
  │ 4. self.compressors: ModuleList[MemoryCompressor]    │
  │    → 每条序列有独立的 MemoryCompressor                │
  │    → 不共享参数，因为不同序列的模式不同               │
  │                                                      │
  │ 5. self.query_seed: Parameter [1, Q, D]              │
  │    → 可学习的初始 query embedding                     │
  │                                                      │
  │ 6. self.query_proj: Linear(2D → D) → GELU → Linear  │
  │    → 将 static_pooled + seq_pooled 投影为 query bias │
  │                                                      │
  │ 7. self.layers: ModuleList[UniScaleLayer]             │
  │    → N 层交替更新 (默认 4 层)                         │
  │                                                      │
  │ 8. self.final_attn: SelfAttentionBlock                │
  │    → 最终融合层                                       │
  │                                                      │
  │ 9. self.fm: FMHead                                    │
  │    → 显式二阶交叉                                     │
  │                                                      │
  │ 10. self.head: LayerNorm → Linear → GELU → Dropout   │
  │     → Linear → 1                                     │
  │     → 最终预测 MLP                                    │
  └──────────────────────────────────────────────────────┘
```

#### `_mean(self, x, mask) → Tensor`

```
输入: x [B, L, D], mask [B, L]
输出: [B, D]

操作: masked mean pooling
  m = mask.float().unsqueeze(-1)        → [B, L, 1]
  return (x × m).sum(dim=1) / m.sum(dim=1).clamp_min(1)

注意 clamp_min(1) 防止除零 (当某个样本的 mask 全 False 时)。
```

#### `forward(self, batch) → dict`

**这是整个模型的核心前向传播。下面是逐步详解。**

```
输入: batch — 来自 TAACCollator 的 dict[str, Tensor]

输出: dict，包含:
  - 'logits': [B] FloatTensor — 未经 sigmoid 的原始输出
  - 'probs':  [B] FloatTensor — sigmoid 后的概率
  - 'aux_similarity': [B] FloatTensor (可选) — 辅助对比分数
```

**Step 1: 静态特征编码**

```python
st = self.tokenizer.encode_static(
    batch['static_token_ids'],      # [B, S]
    batch['static_feature_ids'],    # [B, S]
    batch['static_type_ids'],       # [B, S]
    batch['static_float_values'],   # [B, S, F]
)
# st: [B, S, D] — 每个静态特征的初始 embedding

sm = batch['static_mask']          # [B, S] BoolTensor

for blk in self.static_encoder:
    st = blk(st, sm)
# st: [B, S, D] — 经过 self-attention 精炼后的静态 token
# 此时每个静态 token 已经"看过"其他所有静态 token，
# 实现了用户特征与物品特征之间的交叉

st_pool = self._mean(st, sm)       # [B, D] — 静态特征的全局汇总
```

**Step 2: 序列特征编码 + Memory 压缩**

```python
seq = self.tokenizer.encode_sequence_events(
    batch['seq_token_ids'],         # [B, Ns, L, F]
    batch['seq_feature_ids'],       # [B, Ns, L, F]
    batch['seq_type_ids'],          # [B, Ns, L, F]
    batch['seq_pos_ids'],           # [B, Ns, L]
    batch['seq_name_ids'],          # [B, Ns, L]
)
# seq: [B, Ns, L, D] — 每条序列中每个事件的初始 embedding

mems = []      # 收集每条序列的 memory
seq_pools = [] # 收集每条序列的 pooled 表征

for s in range(Ns):                     # Ns = 3 (action/content/item)
    x = seq[:, s]                       # [B, L, D] — 第 s 条序列
    m = batch['seq_mask'][:, s]         # [B, L]

    for blk in self.seq_encoder:
        x = blk(x, m)                  # 序列内 self-attention
    # x: [B, L, D] — 精炼后的序列 token

    mems.append(self.compressors[s](x, m))   # [B, M, D] — 压缩为 M 个 memory
    seq_pools.append(self._mean(x, m))       # [B, D]
```

**Step 3: 构建 Memory Bank**

```python
mem = torch.cat(mems, dim=1)
# mem: [B, Ns*M, D] — 所有序列的 memory 合并
# 例: 3 sequence × 12 memory = 36 个 memory token

mem_mask = ones([B, Ns*M])
# 全为 True，因为 memory token 是模型自己生成的，不存在 padding

seq_pool = stack(seq_pools, dim=1).mean(dim=1)
# seq_pool: [B, D] — 所有序列的全局汇总
```

**Step 4: Query 初始化**

```python
queries = self.query_seed.expand(B, -1, -1)
# queries: [B, Q, D] — Q 个 query 的初始值（来自可学习参数）

pooled = cat([st_pool, seq_pool], dim=-1)
# pooled: [B, 2D] — 全局上下文信息

query_bias = self.query_proj(pooled).unsqueeze(1)
# query_bias: [B, 1, D] — 基于上下文的偏移

queries = queries + query_bias
# queries: [B, Q, D] — 初始化完成
# 每个 query 既有"通用基底"(seed)又有"样本特定偏移"(bias)
# 这就是 "target-aware" 的 query 初始化
```

**为什么这样初始化 query：**

```
query = 可学习的通用 seed + 每样本上下文 bias

这确保了:
1. 通用 seed 学到不同 query 应关注什么类型的信息
2. 上下文 bias 让每个样本的 query 起点不同
3. 避免了纯零初始化或纯随机初始化的不稳定性
```

**Step 5: N 层交替更新**

```python
for layer in self.layers:          # n_layers 次
    queries = layer(queries, mem, mem_mask, st)
    # 每层内部:
    # 1. queries → cross-attention → memory_bank (提取历史兴趣)
    # 2. cat(queries, static) → TokenMixer (交换信息)
    # 3. → FeedForward (非线性变换)
    # 4. 取出 query 部分作为下一层输入
```

**直觉理解 N 层交替更新：**

```
第 1 层: query 初次接触历史行为 → 建立粗粒度兴趣
第 2 层: query 带着粗粒度兴趣，和静态特征交互 → 初步交叉
第 3 层: 更精细地从历史中提取 → 细粒度兴趣
第 4 层: 最终精炼 → 综合所有信息的 query 表征
```

**Step 6: 最终融合 + 预测**

```python
# 将 query 和 static 拼接做最终融合
fused = cat([queries, st], dim=1)       # [B, Q+S, D]
fused_mask = cat([ones(B,Q), sm], dim=1)
fused = self.final_attn(fused, fused_mask)
# fused: [B, Q+S, D] — query 和 static 做最后一次全局 self-attention

# 多视角 pooling
query_pool = queries.mean(dim=1)         # [B, D] — query 的均值
fused_pool = self._mean(fused, fused_mask)  # [B, D] — 融合后的均值

# FM 显式交叉
fm = self.fm(cat([queries, st], dim=1))  # [B, 1] — 二阶交叉项

# 拼接所有表征送入最终 MLP
final = cat([query_pool, st_pool, fused_pool, fm], dim=-1)
# final: [B, 3*D + 1]
# 包含:
#   query_pool  [D] — 序列兴趣的表征
#   st_pool     [D] — 静态特征的全局表征
#   fused_pool  [D] — 融合后的全局表征
#   fm          [1] — 显式二阶交叉

logit = self.head(final).squeeze(-1)     # [B]
prob = sigmoid(logit)                     # [B]
```

**Step 7: 辅助对比损失（可选）**

```python
if self.use_aux:
    item_embed = self.tokenizer.token_emb(batch['static_token_ids'][:, 1])
    # item_id 固定在静态 token 的位置 1
    # 直接取它的 embedding 作为 item 的原始表征

    aux_sim = cosine_similarity(query_pool, item_embed, dim=-1)
    # aux_sim: [B] — query 表征与 item embedding 的余弦相似度
```

**辅助对比损失的直觉：**

```
如果用户会点击这个 item (target=1):
  → query_pool 应该和 item_embed 方向一致 (sim ≈ +1)

如果用户不会点击 (target=0):
  → query_pool 应该和 item_embed 方向不一致 (sim ≈ -1)

这个辅助损失迫使 query 表征和 item 表征在向量空间中对齐，
增强了模型的表征学习能力。类似于对比学习 (contrastive learning)。
```

#### `compute_loss(self, batch, output) → dict`

```
输入:
  - batch: dict — 包含 'targets' [B]
  - output: dict — forward() 的返回值

输出: dict，包含:
  - 'loss': Tensor — 总损失 (用于反向传播)
  - 'bce_loss': Tensor — BCE 损失值 (detached，仅记录)
  - 'aux_loss': Tensor — 辅助损失值 (detached，仅记录)

操作:
  1. bce = BCE_with_logits(logits, targets)
     → 主损失：二元交叉熵

  2. 若启用辅助损失:
     target_sim = targets × 2 - 1    → 将 {0, 1} 映射到 {-1, +1}
     aux = MSE(aux_similarity, target_sim)
     loss = bce + λ × aux            → λ 默认 0.05

  3. 返回 loss dict

损失函数的数学形式:
  L = -1/N Σ [yᵢ log σ(zᵢ) + (1-yᵢ) log(1-σ(zᵢ))]
      + λ × 1/N Σ (cos(qᵢ, eᵢ) - (2yᵢ-1))²

  其中:
    zᵢ = logit, yᵢ = target, σ = sigmoid
    qᵢ = query_pool, eᵢ = item_embed
```

### 关键训练技术

**混合精度训练 (AMP)**:

```
with torch.amp.autocast('cuda', enabled=use_amp):
    output = model(batch)
    loss = model.compute_loss(batch, output)['loss']
scaler.scale(loss).backward()
scaler.step(optimizer)
scaler.update()

原理:
  - 前向传播用 FP16 (半精度)：更快，省显存
  - 权重更新用 FP32 (全精度)：保证精度
  - GradScaler 防止 FP16 梯度下溢
  - 对 AUC 几乎无影响，训练速度提升 ~1.5x
```

**梯度裁剪**:

```
torch.nn.utils.clip_grad_norm_(model.parameters(), grad_clip)

作用: 防止梯度爆炸
原理: 若 ‖∇L‖ > clip_value，则 ∇L ← ∇L × clip_value / ‖∇L‖
推荐值: 1.0
```

Scaling Law 实验脚本。

```

     a. 计算 params_proxy = d_model × n_layers × num_queries
        （这是模型容量的近似度量）
     b. 读取 history.json → 找到最优 AUC
  2. 拟合 log-linear 关系:
     AUC = a + b × log₁₀(params_proxy)
  3. 输出拟合结果

期望结果:
  slope > 0 → 模型越大，AUC 越高 (Scale Law 成立)
  slope 越大 → 架构的 scaling 效率越高

拟合公式:
  AUC ≈ intercept + slope × log₁₀(d_model × n_layers × num_queries)
```

# 第七部分：设计决策与理论依据

## 7.1 为什么用哈希 Embedding 而不是 Lookup Table

| 比较    | Lookup Table | Hash Embedding           |
| ----- | ------------ | ------------------------ |
| 需要预处理 | ✅ 需遍历全数据建词表  | ❌ 不需要                    |
| 内存    | 与词表大小成正比     | 固定 (hash_size × d_model) |
| 新值    | 需要 OOV 处理    | 自动处理                     |
| 冲突    | 无            | 极低 (4M桶/10万特征≈0.025)     |

在比赛场景下，特征值集合未知且可能很大，哈希 embedding 是最务实的选择。

## 7.2 为什么 Memory 压缩而不是直接 Attention

假设序列长度 L=128，静态 token 数 S=128，query 数 Q=12：

| 方案                          | 复杂度                             | 估算 (D=192) |
| --------------------------- | ------------------------------- | ---------- |
| 全 self-attention on (S+L)×3 | O((S+3L)²D)                     | ~120M ops  |
| Memory 压缩 (M=12)            | O(3×M×L×D) + O(Q×3M×D)×n_layers | ~8M ops    |

15 倍的计算节省，对比赛的时延约束至关重要。

## 7.3 为什么分序列建模

HyFormer 论文的实验表明：

```
简单拼接 3 条序列做统一 self-attention → AUC = X
分序列建模 + query 层面交互            → AUC = X + 0.003~0.005
```

原因：不同序列的语义空间不同（点击 vs 搜索 vs 收藏），简单拼接会让 attention 在异质 token 之间产生虚假的高分数。分序列建模保留了结构信息。

## 7.4 Query 为什么比 [CLS] token 好

传统做法用一个 [CLS] token 放在序列最前面，最后取它的输出做预测。问题：

* [CLS] 只有 1 个 token，信息瓶颈严重
* [CLS] 无法同时关注多个不同的兴趣面

UniScaleFormer 使用 Q 个 query token (Q=12)：

* 每个 query 可以学会关注不同方面
* 最终 mean pooling 保留了多面性
* SelfAttention + TokenMixer 让 query 之间也能交互
* 这是 Perceiver / Querying Transformer 的核心思想

## 7.5 为什么辅助对比损失有效

主损失 (BCE) 只约束最终 logit → target。
辅助损失约束中间表征 → item embedding。

```
只有 BCE:
  模型可能学到 shortcut，如仅用一两个强特征预测
  → 中间 query 表征可能是 degenerate 的

加了辅助损失:
  query 表征必须和 item 表征在向量空间中对齐
  → 迫使 query 学到有意义的兴趣表征
  → 正则化效果 → 泛化能力↑
```

# 第八部分：Scaling Law 理论与实验设计

## 什么是 Scaling Law

在 NLP 领域 (Kaplan et al., 2020)，模型性能 L 与参数量 N 之间存在幂律关系：

```
L(N) = a × N^{-α} + L∞
```

其中 L∞ 是不可约的最优损失。取 log 后为线性关系。

在推荐领域，我们用 AUC 替代 loss (方向相反)：

```
AUC(N) ≈ intercept + slope × log₁₀(N_proxy)
```

## scale_law.py 的设计

**params_proxy = d_model × n_layers × num_queries**

这个代理指标近似反映了模型的"有效容量"：

* d_model: 每个 token 的表征能力
* n_layers: 特征交互的深度
* num_queries: 从序列中提取的信息丰富度

三者的乘积粗略反映了 Transformer 核心部分的参数量。

## 实验建议

1. 用 small/base/large 三套配置训练
2. 运行 `python scale_law.py --configs configs/small.yaml configs/base.yaml configs/large.yaml`
3. 若 slope > 0 → Scaling Law 成立 → 增大模型有意义
4. 若 slope ≈ 0 → 数据量不足以支撑更大模型 → 需要更多数据或更强正则
5. 可以进一步添加 `xlarge.yaml` 等配置，验证幂律是否持续

# 附录 B：常见问题

## Q1: 如果比赛数据没有 content_seq 或 item_seq 怎么办？

修改配置中的 `sequence_names` 即可：

```yaml
sequence_names: [action_seq]  # 只使用一条序列
```

模型会自动调整 `num_sequences`，不需要改代码。

## Q2: 如何增大模型以追求更高 AUC？

按以下优先级增大：

1. `d_model`: 128 → 192 → 256 → 384
2. `n_layers`: 3 → 4 → 6 → 8
3. `num_queries / memory_tokens`: 8 → 12 → 16 → 24
4. `seq_layers / static_layers`: 1 → 2

同时减小 `batch_size` 和 `lr`。

## Q3: 如何处理过拟合？

1. 增大 `dropout`: 0.10 → 0.15 → 0.20
2. 增大 `weight_decay`: 0.02 → 0.05
3. 减小模型大小
4. 使用 `train_limit` 确认不是数据太少

## Q4: item_id 固定在位置 1 有什么影响？

如果你修改了 `_build_static` 中 token 的顺序，需要同步修改 `forward` 中辅助损失取 item embedding 的索引。当前代码中是 `batch['static_token_ids'][:, 1]`。
